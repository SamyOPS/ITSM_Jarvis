1.3  Pourquoi  on  garde  uniquement  ces  fonctionnalités  
1.3.1  Justiﬁcation  “MUST  =  gestion  complète  d’un  ticket”

La  valeur  immédiate  pour  une  PME  repose  sur  la  capacité  à  :  
●  structurer  les  demandes  ●  suivre  leur  traitement  ●  identiﬁer  les  retards  
Le  ticketing  complet  constitue  donc  le  noyau  indispensable.  
1.3.2  Justiﬁcation  “SLA  simple”  
Un  système  de  ticketing  sans  notion  de  délai  ne  permet  pas  de  mesurer  la  performance  ni  
de
 
prioriser
 
efﬁcacement.
 
Toutefois,  la  gestion  avancée  de  SLA  (horaires  ouvrés  complexes,  contrats  multiples)  est  
volontairement
 
exclue
 
aﬁn
 
de
 
limiter
 
la
 
complexité
 
initiale.
 
1.3.3  Justiﬁcation  “Escalade  fonctionnelle”  
L’escalade  introduit  une  logique  proactive  et  différenciante.  
Elle  permet  d’éviter  l’oubli  ou  la  stagnation  d’un  ticket  en  dépassement.  
La  version  1  retient  une  escalade  simple  (niveau  unique),  sans  mécanisme  hiérarchique  
complexe.
 
1.3.4  Justiﬁcation  “CMDB  simpliﬁée”  
26  

---PAGE---

 
La  gestion  de  parc  est  un  élément  différenciant  important  face  à  des  solutions  de  ticketing  
minimalistes.
 
Cependant  :  
●  pas  de  dépendances  techniques  entre  actifs  ●  pas  d’historique  complet  d’affectation  ●  pas  de  cartographie  d’infrastructure  
La  CMDB  est  adaptée  à  un  usage  PME  centré  sur  la  traçabilité.  
1.3.5  Justiﬁcation  du  module  IA  
L’IA  améliore  la  productivité  mais  ne  constitue  pas  une  dépendance  critique.  
Le  système  reste  pleinement  fonctionnel  en  cas  d’indisponibilité  du  service  externe.

Toutes  les  tâches  ne  peuvent  pas  être  réalisées  en  parallèle.  
 
Nous
 
distinguons
 
trois
 
catégories
 
:
 
Tâches  fondations  (non  parallélisables)  
Ce  sont  des  prérequis  techniques  indispensables.  
Exemples  dans  TikAI  :  
●  Mise  en  place  du  socle  technique  (P0)  ●  Conﬁguration  des  environnements  ●  Mise  en  place  du  projet  Supabase  ●  Déﬁnition  du  modèle  de  base  Ticket  
Sans  ces  éléments,  les  autres  développements  seraient  instables.  
Elles  sont  volontairement  traitées  de  manière  séquentielle  pour  limiter  les  risques  
d’architecture
 
divergente.
 
Tâches  dépendantes  d’un  contrat  stabilisé  
Certaines  tâches  peuvent  être  parallélisées  à  partir  du  moment  où  le  contrat  API  est  
déﬁni
,
 
même
 
si
 
l’implémentation
 
complète
 
n’est
 
pas
 
ﬁnalisée.
 
Exemple  :  
●  Le  Back  déﬁnit  le  DTO  Ticket  +  endpoint  GET  /tickets/:id  ●  Le  Front  peut  commencer  à  développer  la  page  “Détail  Ticket”  en  utilisant  des  
données
 
mockées
 
Cela  permet  de  réduire  les  temps  morts.  
Tâches  indépendantes  par  domaine  
Certains  blocs  fonctionnels  sont  sufﬁsamment  isolés  pour  être  développés  en  parallèle.  
Dans  TikAI  :  
●  P2  Référentiels  ●  P3  Tickets  ●  P6  IA  (en  partie)  ●  P5  SLA  (après  création  Ticket)

8.4.2  Séquencement  global  logique  
Le  développement  suit  une  logique  structurée  :  
Phase  1  –  Stabilisation  du  socle  (P0)  
Non  parallélisable  :  
●  Setup  back  ●  Setup  front  ●  Setup  Supabase  ●  CI/CD  ●  Test  “Hello  API”  
Objectif  :  environnement  stable.  
Phase  2  –  Sécurisation  des  accès  (P1)  
Peut  démarrer  dès  que  P0  est  validé  :  
●  Intégration  Supabase  Auth  ●  Guards  RBAC  ●  Protection  endpoints  
L’auth  est  un  prérequis  pour  éviter  de  coder  des  routes  ouvertes  puis  les  sécuriser  plus  tard.  
Phase  3  –  Modèle  métier  central  (Ticket  minimal)  
Avant  toute  chose  :  
●  Entité  Ticket  ●  Statuts  ●  Création  simple  ticket  ●  Endpoint  basique  ●  Persistance  DB  
C’est  la  colonne  vertébrale  du  projet.  
À  partir  de  là,  on  peut  paralléliser.
